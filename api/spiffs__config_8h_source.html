106 // Garbage collecting examines all pages in a block which and sums up
107 // to a block score. Deleted pages normally gives positive score and
108 // used pages normally gives a negative score (as these must be moved).
109 // To have a fair wear-leveling, the erase age is also included in score,
110 // whose factor normally is the most positive.
111 // The larger the score, the more likely it is that the block will
112 // picked for garbage collection.

222 // Enable this to add a temporal file cache using the fd buffer.
223 // The effects of the cache is that SPIFFS_open will find the file faster in
224 // certain cases. It will make it a lot easier for spiffs to find files
225 // opened frequently, reducing number of readings from the spi flash for
226 // finding those files.
227 // This will grow each fd by 6 bytes. If your files are opened in patterns
228 // with a degree of temporal locality, the system is optimized.
229 // Examples can be letting spiffs serve web content, where one file is the css.
230 // The css is accessed for each html file that is opened, meaning it is
231 // accessed almost every second time a file is opened. Another example could be
232 // a log file that is often opened, written, and closed.
233 // The size of the cache is number of given file descriptors, as it piggybacks
234 // on the fd update mechanism. The cache lives in the closed file descriptors.
235 // When closed, the fd know the whereabouts of the file. Instead of forgetting
236 // this, the temporal cache will keep handling updates to that file even if the
237 // fd is closed. If the file is opened again, the location of the file is found
238 // directly. If all available descriptors become opened, all cache memory is
239 // lost.
240 #ifndef SPIFFS_TEMPORAL_FD_CACHE
241 #define SPIFFS_TEMPORAL_FD_CACHE 1
242 #endif
243 
244 // Temporal file cache hit score. Each time a file is opened, all cached files
245 // will lose one point. If the opened file is found in cache, that entry will
246 // gain SPIFFS_TEMPORAL_CACHE_HIT_SCORE points. One can experiment with this
247 // value for the specific access patterns of the application. However, it must
248 // be between 1 (no gain for hitting a cached entry often) and 255.
249 #ifndef SPIFFS_TEMPORAL_CACHE_HIT_SCORE
250 #define SPIFFS_TEMPORAL_CACHE_HIT_SCORE 4
251 #endif

253 // Enable to be able to map object indices to memory.

255 // large files and when changing file offset by seeking around a lot.
256 // When mapping a file's index, the file system will be scanned for index pages
257 // and the info will be put in memory provided by user. When reading, the
258 // memory map can be looked up instead of searching for index pages on the
259 // medium. This way, user can trade memory against performance.
260 // Whole, parts of, or future parts not being written yet can be mapped. The
261 // memory array will be owned by spiffs and updated accordingly during garbage
262 // collecting or when modifying the indices. The latter is invoked by when the

264 // descriptor.
